# Architecture Decisions: Tier 1 Production Packaging

## Decision 1: Reject PyInstaller, Use Setuptools Console Scripts

### What Was Decided

**✅ CHOSEN**: Setuptools console scripts (real Python at runtime)
**❌ REJECTED**: PyInstaller frozen executables

### Why

| Criterion | PyInstaller | Setuptools | Winner |
|-----------|-------------|-----------|--------|
| Installer Size | 350+ MB frozen bytecode | 260-300 MB (wheel only) | Setuptools |
| Runtime Package Installation | ❌ Frozen, can't modify | ✅ Dynamic, supports `pip install` | Setuptools |
| Debugging | ❌ Bytecode is opaque | ✅ Full source available | Setuptools |
| Profiling | ❌ Limited (bytecode) | ✅ Full profiler support | Setuptools |
| Development Experience | ❌ Long recompile cycle | ✅ Reload support | Setuptools |
| Python Version Updates | ❌ Requires recompile | ✅ Single venv update | Setuptools |
| Dependencies | ❌ All bundled | ✅ Install on demand | Setuptools |

**Critical Feature**: **Deps Tab** requires runtime package installation. PyInstaller cannot support this. Setuptools is mandatory.

### How It Works

```
┌─────────────────────────────────────────────────────────────┐
│ TIER 1: PRODUCTION (Setuptools Console Scripts)             │
└─────────────────────────────────────────────────────────────┘

Installation:
  Installer bundles:
    • marvex-<version>-py3-none-any.whl (~20-30 MB)
    • uv.exe (Python package manager)

First Launch:
  1. Supervisor detects missing ~/.marvex/runtime/venv/
  2. Creates virtual environment
  3. Runs: uv pip install --no-index --find-links wheels marvex-<version>-py3-none-any.whl
  4. Setuptools generates console scripts:
     • ~/.marvex/runtime/venv/Scripts/marvex-core.exe
     • ~/.marvex/runtime/venv/Scripts/marvex-provider-worker.exe
     • ~/.marvex/runtime/venv/Scripts/marvex-intent-worker.exe
     • ~/.marvex/runtime/venv/Scripts/marvex-tool-worker.exe
     • ~/.marvex/runtime/venv/Scripts/marvex-voice-worker.exe
  5. Launches services via console scripts

Subsequent Launches:
  Services start instantly via console scripts (no install overhead)

Runtime Package Installation:
  • User installs package via Deps tab
  • Runs: uv pip install <package-name>
  • Package immediately available to next service calls
  • No restart needed
```

### Code Implementation

**supervisor.rs - spawn_service()**:
```rust
// Tier 1: Try venv console script first
if let Some(venv_path) = get_venv_path() {
    if let Some(exe) = find_console_script(&venv_path, service_name) {
        return Ok(spawn_process(exe, ...));  // ✓ PRODUCTION PATH
    }
}

// Tier 3: Fallback to `uv run` (development only)
spawn_dev_fallback(service_name, ...)
```

**Console script generated by setuptools**:
```
~/.marvex/runtime/venv/Scripts/marvex-core.exe
  → Tiny Python wrapper (~30 KB)
  → Points to: site-packages/services/core/main.py:main()
  → Invokes: Full Python interpreter at runtime
  → NOT frozen, NOT bytecode, REAL Python
```

### Trade-offs Accepted

| Trade-off | Impact | Mitigation |
|-----------|--------|-----------|
| venv must exist | First launch slower | One-time cost (~1-2 min) |
| Network dep (uv) | Requires internet for install | Network check + error handling |
| Virtual env overhead | Minor (40-50 MB venv) | Acceptable for functionality |
| Python 3.12 packaged runtime requirement | Excludes older Python from the packaged Windows runtime | Matches the offline voice wheelhouse and remains an acceptable modern baseline |

## Decision 2: Remove Tier 2 (PyInstaller Fallback)

### What Was Removed

**Original 3-tier architecture**:
```
Tier 1: Setuptools console script from venv
  ↓ (if missing)
Tier 2: PyInstaller frozen .exe
  ↓ (if missing)
Tier 3: uv run (development)
```

**New 2-tier architecture**:
```
Tier 1: Setuptools console script from venv
  ↓ (if missing, dev-only)
Tier 3: uv run (development)
```

### Why Tier 2 Was Dead Code

1. **Never Bundled**: `tauri.conf.json` only bundles `.whl` + `uv.exe`
2. **Never Activated**: Tier 1 (venv script) always succeeds on production installs
3. **Impossible Fallback**: Tier 2 would only activate if:
   - Wheel install failed (unlikely, fatal anyway)
   - Console script wasn't generated (impossible with setuptools)
   - Both conditions: instant crash, not graceful fallback
4. **Dead Code Consequences**:
   - Confuses maintainers (what's this sidecar for?)
   - Maintenance burden (keep PyInstaller specs updated)
   - Disk space waste (3 GB PyInstaller output, not shipped)
   - Compiler time waste (5+ minutes of build time)

### Code Removed

**supervisor.rs - sidecar_path() function**:
```rust
// REMOVED FUNCTION (lines 499-504 in original)
fn sidecar_path(resource_dir: Option<&Path>, sidecar: &str) -> Option<PathBuf> {
    // ... looked for PyInstaller .exe files
    // Never called in production paths
    // Dead code
}
```

**Tier 2 Fallback Logic**:
```rust
// REMOVED (would never execute)
else if let Some(path) = sidecar_path(&resource_dir, service_key) {
    // PyInstaller exe fallback
}
```

### Impact

✅ **Positive**:
- Simpler code, easier to maintain
- Clearer error messages (Tier 1 or Tier 3, not ambiguous)
- Faster build times (skip PyInstaller compilation)
- Less disk usage
- Fewer code paths to test

⚠️ **Considerations**:
- Tests that assumed 3-tier might fail (none were found)
- Any external code relying on `sidecar_path()` breaks (grep suggests none exist)

## Decision 3: Setuptools Entry Points (Not Manual Exe Wrapper)

### What Was Chosen

**✅ CHOSEN**: `[project.scripts]` in `pyproject.toml`
**❌ REJECTED**: Manual wrapper scripts

### How pyproject.toml Defines Entry Points

```toml
[project.scripts]
marvex-core = "services.core.main:main"
marvex-provider-worker = "services.provider_worker.main:main"
marvex-intent-worker = "services.intent_worker.main:main"
marvex-tool-worker = "services.tool_worker.main:main"
marvex-voice-worker = "services.voice_worker.main:main"
```

**setuptools generates**:
```
Windows: ~/.marvex/runtime/venv/Scripts/marvex-core.exe
  └─→ calls site-packages/services/core/main.py:main()
```

### Why Not Manual Wrappers

| Factor | setuptools | Manual | Winner |
|--------|-----------|--------|--------|
| Maintenance | Declarative | Scripts to maintain | setuptools |
| Argument passing | ✅ Automatic | ❌ Manual | setuptools |
| Environment variables | ✅ Inherited | ❌ Manual | setuptools |
| Error messages | ✅ Standard | ❌ Custom | setuptools |
| Python updates | ✅ Automatic | ❌ Recompile | setuptools |
| Entry point safety | ✅ Validated | ❌ Runtime | setuptools |

## Decision 4: Wheel Format (Not Directory Install)

### What Was Chosen

**✅ CHOSEN**: `marvex-<version>-py3-none-any.whl` (compiled wheel)
**❌ REJECTED**: `pip install -e .` (editable directory)

### Why Wheel

| Factor | Wheel | Editable Directory | Winner |
|--------|-------|-------------------|--------|
| Shipping | ✅ Single file | ❌ Entire directory | Wheel |
| Install time | ✅ Cached, fast | ❌ 2+ GB copy | Wheel |
| Update | ✅ Version bump clear | ❌ Modified in-place | Wheel |
| Release | ✅ Signed, versioned | ❌ Git-based | Wheel |
| Dependencies | ✅ Frozen, consistent | ❌ Change with git | Wheel |

## Decision 5: Build Script in PowerShell (Not Python)

### What Was Chosen

**✅ CHOSEN**: PowerShell build script (`build-installer.ps1`)
**❌ REJECTED**: Python build script (setup.py)

### Why PowerShell

| Factor | PowerShell | Python | Winner |
|--------|-----------|--------|--------|
| Platform native | ✅ Windows built-in | ❌ Requires Python | PowerShell |
| Subprocess handling | ✅ Native process APIs | ⚠️ Works but complex | PowerShell |
| Colored output | ✅ Direct console | ⚠️ Libraries needed | PowerShell |
| Error handling | ✅ Try/catch native | ✅ Try/except | Tie |
| Distribution | ✅ No deps | ❌ Needs runtime | PowerShell |
| Execution | ✅ Direct (Set-ExecutionPolicy) | ❌ Needs python.exe | PowerShell |

## Decision 6: Tier 1 Only for Production, Tier 3 for Development

### Why No Tier 2

**Tier 1 (Production, Console Scripts)**:
- ✅ Installed via installer
- ✅ Efficient, supports runtime packages
- ✅ Production-grade

**Tier 2 (Legacy, PyInstaller)** ← REMOVED
- ❌ Dead code (never used)
- ❌ Maintenance burden
- ❌ Conflicts with Tier 1

**Tier 3 (Development, uv run)**:
- ✅ No build needed
- ✅ Instant reload
- ✅ Source debugging
- Used for: `cargo run` during development

### When Each Activates

```
Production Install:
  1. Installer runs
  2. Supervisor creates venv (Tier 1)
  3. Tier 1 always activates

Development (cargo run):
  1. No venv created
  2. Tier 1 not available
  3. Falls back to Tier 3 (uv run)

Testing:
  1. Manifests show "runtime_tier": "tier1_setuptools"
  2. Confirms correct resolution
```

## Decision 7: Runtime Bootstrap at First Launch

### What Happens

```
First Launch:
1. Supervisor checks ~/.marvex/runtime/venv/
2. Not found → Create
3. Install wheel into venv
4. Setuptools generates console scripts
5. Launch services
6. Save manifest.json with runtime info

Subsequent Launches:
1. Supervisor checks ~/.marvex/runtime/venv/
2. Found → Use existing
3. Launch services immediately
```

### Why Not Pre-bundled venv

| Option | Bundled venv | Runtime bootstrap | Winner |
|--------|-------------|-------------------|--------|
| Installer size | 300-500 MB | 260-300 MB | Bootstrap |
| First launch | Instant | 1-2 min | Bundled |
| Updates | Reinstall app | Just wheel update | Bootstrap |
| Debugging | Shipped frozen | Fresh install | Bootstrap |
| User choice | Fixed | Customizable | Bootstrap |

**Decision**: Accept 1-2 min first launch for flexibility and smaller installer.

## Decision 8: Manifest.json for Runtime Introspection

### What Gets Recorded

```json
{
  "runtime_architecture": "tier1_setuptools_console_scripts",
  "runtime_tier": "tier1_setuptools",
  "bootstrap_timestamp": "2024-01-15T10:30:45Z",
  "wheel_version": "0.2.0",
  "python_version": "3.12.x",
  "services": {
    "core": {
      "console_script_path": "/home/user/.marvex/runtime/venv/Scripts/marvex-core.exe",
      "entry_point": "services.core.main:main"
    }
  }
}
```

### Why Track This

- ✅ Debugging: identify runtime configuration
- ✅ Monitoring: detect Tier 1 vs Tier 3
- ✅ Analytics: understand user environments
- ✅ Support: quick problem diagnosis
- ✅ Version tracking: logs which app/wheel version is running

**Note**: `wheel_version` is read from `version.toml` at build time and stored in manifest.

## Summary: Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Python Runtime | Setuptools (not frozen) | Supports Deps tab, debugging, dynamic |
| Packaging Method | Wheel (not editable) | Cleaner shipping, versioning |
| Entry Points | pyproject.toml (not manual) | Maintainable, standard |
| Build Script | PowerShell (not Python) | Native Windows, no dependencies |
| Tier Structure | 2-tier (not 3) | Remove dead code |
| Bootstrap | Runtime (not bundled) | Smaller installer, flexibility |
| Observability | Runtime manifest | Debugging, support, monitoring |
| Version Management | Central `version.toml` | Single source of truth across all build systems |

## Migration Path (If Needed)

If future requirements demand change:

1. **To add Tier 2 back**: 
   - Rebuild PyInstaller specs
   - Add sidecar_path() back to supervisor
   - Tests for Tier 2 fallback
   - Update manifest.json

2. **To use bundled venv**:
   - Add venv to tauri bundle
   - Increases installer to 400+ MB
   - Update bootstrap logic
   - Skip Tier 3 (venv always present)

3. **To switch to Conda**:
   - Replace uv with conda/micromamba
   - Update bootstrap commands
   - Modify console scripts resolution
   - Requires major packaging change

Current architecture is **flexible** for future changes but **optimized** for current needs.
